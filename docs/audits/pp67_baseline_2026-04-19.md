## Key findings

- **Four languages already PASS PP67:** Dutch, German, Korean, Spanish.
- **Italian + Japanese account for 92%** of all PP67 violations (225 / 243). The
  root cause is per-lesson batch density files (`_batch*.js`) that rely on
  `mc` recognition and `match` pairs instead of `fb`/`drag_fill` retrieval.
  The main `unit-NN.js` files are mostly clean.
- **Russian (9/36) and French (5/30)** have tactical gaps in specific units
  with very low production counts (several at 0 production). Remediation is
  one unit at a time.
- **Chinese (1/45)** is a single-unit fix (`unit-41.js`).
- **Portuguese (3/36)** is a single-unit batch — three late-sequence units
  need one additional `fb` or `drag_fill` each.

## Remediation priority

1. **P1 quick wins (low friction):** chinese-v2 unit-41, portuguese-v2 unit-09/22/23,
   russian-v2 unit-17/18/19/22/24/29/35/08/20, french-v2 unit-02/11/12/14/26.

   answer is the target-language form taught in that unit.
2. **P2 larger remediation:** italian-v2 (104 units) and japanese-v2 (121 units).
   Approach: per-batch file edit adding a single `fb` or `drag_fill` that
   uses an already-taught trg.

## Method

1. Build `allTrgs` set by scanning every `teach` block for `trg:` or legacy `nl:`
   field across all unit + batch files.
2. For each quiz step classify as production iff the answer matches a normalized
   `allTrgs` entry (case-insensitive, article-stripped).
3. `match` steps and `mc` steps are intentionally NOT counted as production per
   PP67 wording (PP67: "fb or drag_fill where the answer is the target-language
   form"). Retrieval research (Swain output hypothesis) supports this strictness.
