# Dictionary Expansion Checklist (Per Language)
> Extracted from Korean dictionary build (D117, 2026-03-21).
> Use when adding full dictionary support to any new language.

---

## Required Components (6 items)

### 1. Conjugation Engine
- **What it is**: A `{lang}-conjugation.js` file with a `conjugate{Lang}Verb()` function.
- **What it does**: Returns all conjugated/inflected forms for any verb (and adjective, if applicable). Powers the Forms tab in WordPopup and the form index.
- **Korean example**: `src/data/korean-conjugation.js` (~300 lines). 7 irregular types, ~21 ending templates across speech levels (해요체/합쇼체/반말) x tense. Includes `detectIrregType()`, `nounWithParticles()`.
- **Status for other languages**: German conjugation engine exists (`german-conjugation.js`). Dutch, French, Spanish, Italian, Japanese still need their own engines.

### 2. Function Word List
- **What it is**: A `function-words-{lang}.js` file in `src/data/wordlists/`.
- **What it does**: Seeds WORD_DB with articles, prepositions, conjunctions, pronouns, and particles that do not have teach cards but must be tappable in lessons.
- **Korean example**: `src/data/wordlists/function-words-ko.js` (~560 lines, 531 words).
- **Status for other languages**: Only Korean has this file. All other languages need it before full dictionary coverage is achievable.

### 3. POS Tagger Plugin
- **What it is**: A `POS_TAGGERS["{lang}"]` entry in `src/data/dictionary.js`.
- **What it does**: Resolves POS for words that cannot be inferred from teach-card `pos` fields alone (e.g., context-dependent words, particles, auxiliaries).
- **Korean example**: `POS_TAGGERS["ko"]` — handles particles (~prefix), Sino-Korean compounds, verb stem resolution.
- **Status for other languages**: Korean is the only plugged-in language. Each new language needs a tagger registered before WORD_DB builds correctly for that language.

### 4. Form Index (Reverse Lookup)
- **What it is**: A `{LANG}_FORM_INDEX` map from conjugated/inflected surface forms back to their dictionary lemma.
- **What it does**: Lets the engine recognize "갔어요" as a form of "가다" so tapping any word in a sentence finds the right WORD_DB entry.
- **Korean example**: `KOREAN_FORM_INDEX` in `dictionary.js` — ~6,000+ entries generated by running every lemma through the conjugation engine.
- **Status for other languages**: German has `GERMAN_FORM_INDEX` (wired into `lookupWord()` + `universalHl()`). Dutch, French, Spanish, Italian, Japanese do not have one yet.

### 5. Example Index
- **What it is**: A `{LANG}_EXAMPLE_INDEX` map from each word to all curriculum sentences containing it, grouped by unit.
- **What it does**: Populates the Examples tab in WordPopup, showing learners real sentences from the lessons they are studying.
- **Korean example**: `KOREAN_EXAMPLE_INDEX` in `dictionary.js` — built at module load by scanning all teach-card `example` fields.
- **Status for other languages**: Only Korean has this index. German's example sentences are accessible but not indexed this way yet.

### 6. Grammar Reference
- **What it is**: A `{LANG}_GRAMMAR_REFERENCE` array built from the language's `tip` cards, `verb_table` steps, and `kind:"grammar"` teach cards.
- **What it does**: Populates the Grammar tab on VocabularyPage with searchable, filterable grammar entries (by CEFR level, category, politeness/register).
- **Korean example**: `KOREAN_GRAMMAR_REFERENCE` — 159+ items from 3 sources, 11 categories, politeness detection via `detectPoliteness()`, emoji-stripped titles via `cleanTitle()`.
- **Status for other languages**: German grammar module exists (`src/data/grammar/german.js`, 58 entries). GrammarPage is wired to load from `GRAMMAR_REFS[lang]`. French (52), Spanish (53), Korean (47), Italian (12), Japanese (20) grammar modules exist. Dutch grammar content is mock/placeholder.

## WORD_DB Entry Shape

Base shape (all languages):

```javascript
{
  word: "가다",        // surface form (the key in WORD_DB[lang])
  display: "가다",     // what is shown in the UI
  en: "to go",        // English translation
  pos: "verb",        // POS tag (must match POS_COLORS keys)
  gender: null,       // grammatical gender: "m" | "f" | "n" | "pl" | null
  level: "A1",        // CEFR level from the teach card
  tags: ["verb"],     // category tags array
  taught: true,       // true = has dedicated teach card; false = function word only
  isLemma: true,      // true = appears in vocab browse list
  lemma: "가다",      // base/dictionary form
}
```

TOPIK enrichment fields (Korean only, added in D117):

```javascript
  topikLevel: "A",    // "A" (A1-A2) or "B" (B1-B2) from official TOPIK list
  topikPos: "verb",   // POS as recorded in official TOPIK list
  hanja: null,        // Chinese character form if Sino-Korean
  frequencyRank: 25,  // NIKL corpus rank (badge shown when rank < 3000)
```

Arabic enrichment fields (added 2026-04-08, stored on German WORD_DB entries):

```javascript
  ar: "يذهب",         // Arabic translation of the target word
  noteAr: "...",      // Arabic translation of the teach card note
  funFactAr: "...",   // Arabic translation of the fun fact
  exampleAr: "...",   // Arabic translation of the example sentence
```

When adding a new language's enrichment fields, follow the same pattern: language-code prefix on each field, stored directly on the WORD_DB entry, resolved in the UI via `baseLang`-aware helper functions.

## Coverage Methodology

### Three-Tier Classification

When running a gap audit against an official word list (e.g., TOPIK, Goethe, DELF):

| Tier | Definition | PP55 Status |
|------|-----------|-------------|
| **Taught** | Has a dedicated teach card (`trg`/`nl` field matches the word, stem, or its 하다-compound). | PASS |
| **Partial** | Appears in a phrase, example sentence, note, or deepDive, but has no standalone teach card. | FAIL — must be promoted to a teach card. |
| **Missing** | Not found anywhere in the curriculum. | FAIL — must be added as a new teach card. |

Key rule (from PP52/PP34): "Taught" means its own teach card. Example-only or deepDive-only does NOT count.

### Compound Word Insight

Many "missing" words from official lists actually appear in the curriculum as parts of compound words, in example sentences, or embedded in notes. They still fail PP55 because they lack standalone teach cards. When auditing:
1. Search the raw unit files for the word form AND common stems/conjugations.
2. If found only in an example or compound breakdown, classify as Partial, not Taught.
3. Every Partial becomes a teach card task for the curriculum restructure.

### Coverage Measurement Workflow

1. Import the official word list file into `dictionary.js` (same pattern as `topik-vocab-ko.js`).
2. At module load, run a post-processor that matches each list entry against `WORD_DB[lang]` entries.
3. Tag matches with enrichment fields (`topikLevel`, `frequencyRank`, etc.).
4. Unmatched entries = Missing. Log them to a coverage report file (`docs/{lang}_COVERAGE_REPORT.md`).
5. Check coverage percentages per CEFR level. All levels must reach 100% before the language is PP55-compliant.
6. Re-run after each content build session to track progress.
